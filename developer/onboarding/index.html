    Developer Onboarding Guide¶

    🛠️ Engineering

    Welcome to Remitpro Engineering¶

    This guide gives a new engineer everything they need to clone the repo, understand the codebase, and ship their first change within a single working day. Read it end-to-end before touching any code.

    Time to complete: ~3–4 hours including environment setup.

    1. Project Context¶

    Remitpro is a production multi-tenant SaaS built for Nigerian educational institutions. It automates school fee collection via bank virtual accounts (Monnify), manages academics, HR, payroll, and student records.

    Business Model¶

    Every fee payment → Monnify splits instantly:
  ├── School subaccount  → Remainder (e.g. ₦49,800)
  └── Platform account  → Flat ₦200 fee (our revenue)

    Key Stakeholders¶

    Role Who Description
    Super-Admin Platform owner (Remitpro Ltd.) Sees all schools, monitors revenue
    School Admin / Owner School principal or registrar Full control over one school workspace
    Bursar Finance staff Manages fees, payments, expenses
    Teacher Academic staff Enters grades, marks attendance
    Parent Student's guardian Portal-only; pays fees, views results

    Tool Minimum Version Install

    3. Repository Setup¶

    3.1 Clone & Install¶

    # Clone the repository
git clone git@github.com:remitpro/schoolpay.git
cd schoolpay

# Install all dependencies (frontend + backend are in the same repo)
npm install --legacy-peer-deps

    Legacy Peer Deps Flag

    The --legacy-peer-deps flag is required due to a peer dependency conflict between @sentry/react and the latest React ecosystem. This is tracked in the technical debt register.

    3.2 Environment Variables¶

    Copy the template and fill in your values:

    cp .env.example .env

    Then edit .env. Below is a complete reference:

    .env — Complete Reference
    # ── Frontend (Vite prefix = exposed to browser) ──────────────
# Supabase project URL — used for file storage only
VITE_SUPABASE_URL=https://your-project.supabase.co

# Supabase anon/publishable key (safe to expose)
VITE_SUPABASE_PUBLISHABLE_KEY=eyJhbGci...

# Points the React app at your local Express backend
VITE_MICROSERVICES_URL=http://localhost:5001/api

# ── Backend (Server-side only, never in browser) ─────────────
# Neon PostgreSQL connection string (use the POOLER URL)
DATABASE_URL=postgresql://user:pass@ep-xxx-pooler.neon.tech/neondb?sslmode=require

# Monnify credentials (use SANDBOX values for dev)
MONNIFY_API_KEY=MK_TEST_...
MONNIFY_API_SECRET=...
MONNIFY_API_URL=https://sandbox.monnify.com

# Your platform's Monnify sub-account (₦200 fee destination)
PLATFORM_SUBACCOUNT_CODE=MFY_SUB_...

# Notification services
RESEND_API_KEY=re_...
TERMII_API_KEY=TL...

# Super-admin elevation target email
OWNER_EMAIL=admin@remitpro.io

# JWT secret — MUST be a strong random string in production
JWT_SECRET=change-me-in-production-32chars-min

# Express server port
PORT=5001

    Never Commit .env

    The .env file is listed in .gitignore. Never commit real credentials. Use fly secrets set for production and GitHub Secrets for CI.

    3.3 Database Setup (Dev)¶

    # 1. Push the Drizzle schema to your Neon dev database
npm run db:push

# 2. Apply Row Level Security policies
npm run db:rls

# 3. Create database triggers (auto-profile sync, audit hooks)
npm run db:triggers

# 4. Seed demo data (Greenwood International School)
npm run db:seed

    Use the Dev Branch DB

    The project has two Neon databases:

    • Dev → Used for local development (in .env)
    • Prod → The live system (in .env with the # prod comment)

    Never point your local DATABASE_URL at production.

    3.4 Start the Development Servers¶

    You need two terminal sessions running simultaneously:

    Terminal 1 — Express Backend (Port 5001)
    npm run server:dev
# Expected output:
# 🚀 Microservices Express server running on port 5001
# 📡 Endpoints registered under /api/
    Terminal 2 — Vite Frontend (Port 8080)
    npm run dev
# Expected output:
# ➜  Local:   http://localhost:8080/

    Open http://localhost:8080 in your browser.

    3.5 Verify Your Setup¶

    Click "Sign in as Demo Admin" on the login page. If you see the Greenwood dashboard load, your setup is correct.

    Health Check

    You can also verify the backend directly: 

    curl http://localhost:5001/health
# {"status":"ok","timestamp":"..."}

    5. Critical Concepts You Must Understand¶

    5.1 The Mock Supabase Client (src/integrations/supabase/client.ts)¶

    Most Important File in the Codebase

    This is the #1 thing that confuses new developers. Read this section carefully.

    The project started with Supabase but migrated to a custom Express/Neon backend. Rather than refactoring every component, client.ts intercepts all Supabase API calls and proxies them to the Express server.

    Component calls:          → Actually goes to:
supabase.auth.signIn()    → POST http://localhost:5001/api/auth/login
supabase.auth.signUp()    → POST http://localhost:5001/api/auth/signup
supabase.from('students') → POST http://localhost:5001/api/db-proxy
supabase.functions.invoke → POST http://localhost:5001/api/{function-name}

    What this means for you: - Supabase Auth is NOT used for authentication - All auth is JWT-based via the Express server (server/src/auth-helper.ts) - All DB queries go through the /api/db-proxy route (server/src/routes/dbProxy.ts) - The real Supabase client is only used for file storage (attendance photos, uploads)

    5.2 Authentication Flow¶

    sequenceDiagram
    participant Browser
    participant client.ts as client.ts (Mock)
    participant Express as Express :5001
    participant Neon as Neon PostgreSQL

    Browser->>client.ts: supabase.auth.signInWithPassword({email, password})
    client.ts->>Express: POST /api/auth/login
    Express->>Neon: SELECT * FROM profiles WHERE email = ?
    Neon-->>Express: profile row (with passwordHash)
    Express->>Express: bcrypt.compare(password, hash)
    Express-->>client.ts: { token: "eyJ...", user: {...} }
    client.ts->>Browser: localStorage.setItem('remitpro-session-token', token)
    Browser->>Browser: window.location → subdomain redirect

    The JWT token is stored in localStorage under key remitpro-session-token and sent as Authorization: Bearer <token> on every subsequent API request.

    5.3 Multi-Tenancy & Subdomain Routing¶

    Each school gets a unique subdomain (e.g., greenwood.localhost:8080). The useSchoolSettings hook reads the subdomain from window.location.hostname to determine which school's data to display.

    // src/utils/subdomain.ts
export function slugifySubdomain(text: string): string { ... }
export function buildSubdomainUrl(subdomain: string, path: string, token?: string): string {
  // In dev: greenwood.localhost:8080/?token=...
  // In prod: greenwood.remitpro.io/?token=...
}

    5.4 The DB Proxy (/api/db-proxy)¶

    All supabase.from('tableName').select/insert/update/delete() calls from the frontend go through dbProxy.ts. The proxy:

    1. Reads the JWT token from the Authorization header
    2. Verifies it to get schoolId and role
    3. Automatically scopes all queries to the user's school_id
    4. Executes the query against Neon via Drizzle ORM

    This is the application-level multi-tenancy enforcement. RLS at the PostgreSQL level provides a second layer.

    5.5 The ₦200 Revenue Model¶

    When a student's virtual account receives a transfer, Monnify sends a webhook to /api/monnify-webhook. The server:

    1. Validates the HMAC-SHA512 signature
    2. Deducts ₦200 as platform_fee in the transactions table
    3. Updates the student's ledger (student_ledgers.amount_settled)
    4. Recalculates students.outstanding_balance
    5. Sends a receipt notification

    6. Development Workflows¶

    6.1 Adding a New Page¶

    1. Create src/pages/MyNewPage.tsx
    2. Add the route in src/App.tsx: 
      <Route path="/my-new-page" element={<MyNewPage />} />
    3. Add a nav link in the sidebar component (src/components/layout/Sidebar.tsx)
    4. Gate it with role if needed: 
      {role === 'admin' && <Route path="/admin-only" element={<AdminPage />} />}

    6.2 Adding a New API Endpoint¶

    1. Create server/src/routes/myNewRoute.ts: 

      import { Router } from 'express';
import { authMiddleware } from '../auth-helper';
import { db } from '../db';

const router = Router();

router.post('/my-endpoint', authMiddleware, async (req, res) => {
  try {
    const { schoolId } = req.user!;
    // ... query Neon via Drizzle
    return res.json({ success: true });
  } catch (err: any) {
    return res.status(500).json({ error: err.message });
  }
});

export default router;

    2. Register it in server/src/index.ts: 

      import myNewRouter from './routes/myNewRoute';
app.use('/api', myNewRouter);

    6.3 Adding a New Database Table¶

    1. Add the table definition to src/db/schema.ts (Drizzle ORM)
    2. Push the schema change: 
      npm run db:push
    3. If the table needs RLS, add a policy in src/db/setup_rls.ts and run: 
      npm run db:rls
    4. Export the table in the import lists used in dbProxy.ts and seed.ts

    Migration vs Push

    db:push applies schema changes directly (dev-friendly). For production, generate migrations first: 

    npm run db:generate  # Creates SQL in /drizzle/
# Review the generated SQL, then:
npm run db:push

    6.4 Working with the Service Layer¶

    Data access logic lives in src/services/. Always use these instead of calling supabase.from() directly in components:

    // ✅ Correct — use the service
import { FeeService } from '@/services/FeeService';
const ledger = await FeeService.generateLedgerEntries(studentId, class, term);

// ⛔ Avoid — direct DB call in component
const { data } = await supabase.from('student_ledgers').insert({...});

    8. Architecture Decision Records (ADRs)¶

    These are key decisions made before you joined. Understanding them prevents re-litigating solved problems.

    ADR-001: Custom Auth over Supabase Auth¶

    Decision: Replace Supabase GoTrue authentication with a custom JWT system (bcrypt + jsonwebtoken).

    Reason: Supabase Auth does not natively support the multi-tenant subdomain model where a school_id must be embedded in every JWT claim. The custom system gives full control over the token payload.

    Impact: supabase.auth.* calls in the UI are intercepted by client.ts and proxied to Express.

    ADR-002: Express Backend over Supabase Edge Functions¶

    Decision: Migrate all Supabase Edge Functions (Deno) to a Node.js/Express server deployed on Fly.io.

    Reason: Deno Edge Functions had cold-start latency, limited npm support, and poor debugging tooling. Node/Express with Drizzle gives full TypeScript support, local debugging, and direct DB access.

    Impact: supabase.functions.invoke('name', ...) calls are intercepted by client.ts and proxied to http://localhost:5001/api/{name}.

    ADR-003: Neon PostgreSQL over Supabase Postgres¶

    Decision: Use Neon as the primary database, with Supabase used only for storage and the legacy client interface.

    Reason: Neon provides branching, autoscaling, and a better pricing model for multi-tenant workloads. Supabase PostgREST was replaced by the DB proxy to maintain query control.

    ADR-004: Drizzle ORM over Prisma¶

    Decision: Use Drizzle ORM for all database access.

    Reason: Drizzle generates zero overhead at runtime, has superior TypeScript inference, and produces readable SQL. Prisma's abstraction layer and migration model were incompatible with direct Neon + RLS workflows.

    ADR-005: Application-Level Tenancy via DB Proxy¶

    Decision: All CRUD queries are scoped to school_id at the application layer (dbProxy.ts), backed by PostgreSQL RLS as a second enforcement layer.

    Reason: Pure RLS requires passing auth.uid() through the connection, which doesn't work cleanly with direct Drizzle connections. The two-layer approach (app + DB) provides defense-in-depth.

    9. Branching & Code Review¶

    main        ← production-ready code (protected)
dev         ← integration branch (merge features here first)
feat/*      ← feature branches (branch from dev)
fix/*       ← bug fix branches
hotfix/*    ← urgent prod fixes (branch from main)

    Pull Request Process¶

    1. Branch from dev: git checkout -b feat/my-feature dev
    2. Write code & tests
    3. Run npm run lint && npm run test locally
    4. Open PR → dev (never directly to main)
    5. Request review from at least 1 team member
    6. Squash merge after approval

    Commit Message Convention¶

    type(scope): short description

feat(finance): add partial payment support to ledger
fix(auth): handle expired JWT in session middleware
docs(api): add demo-login endpoint reference
chore(deps): upgrade drizzle-kit to 0.31.10

    Types: feat, fix, docs, style, refactor, test, chore

    11. Performance & Observability¶

    Key Metrics to Monitor¶

    Metric Where to Check
    Backend errors fly logs --app remitpro-backend
    DB query performance Neon Console → Query analytics
    Frontend errors Sentry (configured via @sentry/react)
    Uptime UptimeRobot / Pingdom
    Payment webhooks Monnify Dashboard → Webhook logs

    Common Performance Traps¶

    • N+1 queries in the DB proxy: The dbProxy.ts handles one table at a time. Complex joins should be handled in dedicated Express routes, not through the proxy.
    • Large CSV imports: Student bulk imports are processed synchronously. For >500 students, this can time out. Future: move to a background job.
    • Attendance photo uploads: Photos go to Supabase Storage. Keep bucket policies correct or uploads will silently fail.

    12. Security Checklist for New Features¶

    Before merging any PR, verify:

    • New API routes use authMiddleware (never skip auth on non-demo routes)
    • DB queries are scoped by school_id from req.user.schoolId
    • No secrets are logged to console.log
    • Input is validated (use Zod schemas)
    • New tables have RLS policies added to setup_rls.ts
    • Sensitive mutations call AuditService.log()
    • CORS is not widened beyond * without documented reason
