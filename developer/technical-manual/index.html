    Remitpro: Deep Technical Manual 🛠️¶

    This document provides architects and developers with a deep-dive into the internal workings of the Remitpro platform.

    🏗️ 1. Multi-Tenant Architecture¶

    Logical Isolation (Row Level Security)¶

    Remitpro uses a shared-database, shared-schema multi-tenant model. Isolation is strictly enforced at the database level using PostgreSQL Row Level Security (RLS) on Neon.

    • Global Context: The auth.uid() of the requester is used to fetch the school_id from the profiles table.
    • Enforcement: Most tables have an ENABLE ROW LEVEL SECURITY command applied.
    • Policies: 
      CREATE POLICY school_isolation_policy ON students 
USING (school_id = (SELECT school_id FROM profiles WHERE id = auth.uid()));
    • Super-Admin Bypass: Super-admins have a distinct role in the profiles table that allows them to bypass RLS for platform-wide monitoring.

    2. Database Schema (Drizzle ORM)¶

    The schema is designed for high financial integrity and auditability. Key entities include:

    • Schools: Primary tenant entity. Stores Monnify sub-account codes and academic cycle settings.
    • Profiles: Extended user data. Links individual Auth users to a school.
    • Master Fee Schedule: The "Rulebook" for a school. Defines what each class (e.g., JSS1) should pay.
    • Student Ledgers: The "Debt Tracker". When a term starts, entries are generated here based on the Master Schedule.
    • Transactions: Immutable records of every Naira that enters the system.

    💰 3. The "₦200 Flat-Fee" Revenue Model¶

    Remitpro implements a unique revenue model where the platform owner earns ₦200 on every transaction.

    Implementation Logic¶

    The split is handled during the Monnify Transaction Initialization: 1. Request: The system requests a transaction for ₦10,000. 2. Split Rule: - Platform Subaccount: ₦200 (Fixed) - School Subaccount: ₦9,800 (Remainder) 3. Settlement: Monnify handles the split instantly upon payment.

    Note: This is managed within the generate-virtual-account and monnify-webhook microservice routes.

    ⚡ 4. Microservices (Node/Express Server) & Integrations¶

    The backend backend services run on a Node/Express server (located in server/) written in TypeScript, connecting directly to the Neon PostgreSQL database using Drizzle ORM. Frontend invocations of supabase.functions.invoke(...) are intercepted in client.ts and redirected to http://localhost:5001/api/<function-name>.

    onboard-school¶

    • Trigger: Called manually from the Onboarding UI.
    • Route: POST /api/onboard-school
    • Logic: Creates the schools record in Neon and asynchronously calls Monnify to provision a subAccountCode. Links the current user as the admin of that school.

    generate-virtual-account¶

    • Trigger: Triggered when a staff member creates a virtual account for a student.
    • Route: POST /api/generate-virtual-account
    • Logic: Provisions a reserved account (Wema/Rolez) via Monnify. The account is permanently mapped to the student's studentId and saved to Neon.

    monnify-webhook¶

    • Trigger: Incoming HTTP POST from Monnify.
    • Route: POST /api/monnify-webhook
    • Logic:
    • Validates the Monnify Hash Signature.
    • Identifies the student via the productCode or transactionReference.
    • Updates the student_ledgers directly in Neon (marking debts as paid).
    • Records the transactions entry for auditing.

    verify-payment¶

    • Trigger: Client-side widget callback or manual admin check.
    • Route: POST /api/verify-payment
    • Logic: Queries Monnify API directly (Server-to-Server) to confirm status of a transaction reference. Updates the Neon DB directly only if valid. Sends receipt via send-email.

    invite-user¶

    • Trigger: Admin sends invitation.
    • Route: POST /api/invite-user
    • Logic: Handles Supabase Auth invitation via the administrative SDK, assigns multi-tenant schoolId and role in Neon, and sends branded onboarding email via Resend.

    send-email¶

    • Trigger: Internal Service Call (supabase.functions.invoke).
    • Route: POST /api/send-email
    • Logic: Proxies requests to Resend API. Keeps API keys hidden from client. Log-only mode if keys missing.

    promote-students¶

    • Trigger: Admin "End Session" action.
    • Route: POST /api/promote-students
    • Logic:
    • Batches students by class.
    • Updates class string (e.g., JSS1 -> JSS2) in Neon.
    • Archives current academic records for historical reporting.
    • Resets currentTerm to TERM_1.

    🏗️ 5. Smart Infrastructure (Operations & Scale)¶

    5.1 Resilience & Offline Sync¶

    The platform implements an Offline-First strategy for high-frequency operations: - Service Worker (PWA): Uses vite-plugin-pwa to cache assets and core UI, allowing the app to launch without internet. - Attendance Queueing: Uses OfflineStore (LocalStorage) to capture student roll-calls. - Background Sync: A useEffect hook monitors the online window event. Upon reconnection, it automatically upserts the local queue to Neon and clears the cache.

    5.2 Biometric Clock-In Logic¶

    • Geofencing: Utilizes geolib to compare the teacher's current coordinates against the school's configured geofence_lat/lng. Access to the camera is only granted if the distance is within the geofence_radius.
    • Liveness Verification: Implemented a multi-state machine for biometric capture, requiring the user to smile/blink to generate the storage-ready attendance photo.

    5.3 Institutional Ecosystem Services¶

    • CurriculumService: Manages the versioned repository of academic documents. Documents are stored in protected Supabase buckets with RLS metadata.
    • AnnouncementService: Centralized engine for class-based and school-wide broadcasts.
    • QuizService: Handles the logic for Computer Based Testing, including randomized question rendering and automated result computation.

    🎨 6. Design System & Theming¶

    Indigo-Night Design System¶

    The platform utilizes a custom-built design system based on: - Primary Color: Modern Indigo (hsl(243 75% 59%)). - Surface Logic: Glassmorphism (backdrop-blur-xl) with high-transparency card backgrounds. - Dark Mode: Persisted using next-themes and the class attribute on the <html> element.

    State-of-the-Art Visuals¶

    • Mesh Gradients: Used in the Login/Onboarding background for professional "Fintech" depth.
    • Dynamic Logo Inversion: The .brand-logo utility automatically inverts logo colors for dark backgrounds using CSS filters.

    📈 7. Audit & Compliance¶

    • Audit Logs: Sensitive actions (fee changes, manual payment records) are captured in the audit_logs table.
    • Immutability: Once a transaction is recorded, it cannot be deleted or modified through the application UI.

    🛡️ 8. Type Safety & Error Handling¶

    Starting with v1.2, the platform enforces strict TypeScript conventions to eliminate "runtime surprises."

    8.1 Specific Typing over any¶

    • Data Entities: Components use Supabase-generated Tables types from @/integrations/supabase/types.
    • Service Methods: All service methods (e.g., PayrollService, StaffService) are strictly typed to ensure predictable data flow.
    • Mutations: React Query mutations use typeof formData or specific interfaces to validate inputs before submission.

    8.2 Robust Error Boundaries¶

    All asynchronous operations use a standardized error-catching pattern: 

    try {
  // async operation
} catch (err: unknown) {
  const message = err instanceof Error ? err.message : "A persistent error occurred";
  toast.error(message);
}
    This ensures that the user always receives a human-readable explanation of failures, and the system never silently crashes due to malformed error objects.
